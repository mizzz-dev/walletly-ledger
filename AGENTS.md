# AGENTS.md

## 1. このプロジェクトの目的
このリポジトリは、**同棲・ルームシェア・家族・仕事**で共通利用できる「共有家計簿 / 簡易会計アプリ」を継続的に育てるための実装基盤です。  
短期的には日々の支出管理を、長期的には OCR・銀行連携・税務対応まで見据えた拡張を前提にしています。

---

## 2. プロダクト概要
- 対象: 複数メンバーでの家計・経費管理
- 現在の主機能:
  - `Household / Ledger` 解決（Authセッション起点）
  - カテゴリ・メンバーシップ・プリセット管理
  - 支出登録（`/transactions/new`）
  - 精算提案・精算記録（`/settlements`）
  - 管理画面（`/admin/presets`）
- 今後の拡張:
  - 取引の保存/一覧強化、ダッシュボード
  - OCR、銀行連携、税務対応

---

## 3. 技術スタック
- Next.js App Router
- TypeScript
- Tailwind CSS
- shadcn/ui
- Supabase（Auth / Postgres / RLS / Storage想定）
- PWA
- Vitest

---

## 4. ドメインの重要概念
- **Household**: 利用単位（同居・家族・チーム）。アクセス制御の最上位スコープ。
- **Ledger**: Household配下の台帳。用途別（生活費/旅行/業務等）で分離可能。
- **Membership**: ユーザーとHouseholdの関係。`owner / editor / viewer` の権限基盤。
- **Category**: 支出分類。集計・分析・税務連携の基礎。
- **Transaction**: 支出/収入の記録本体。
- **Split**: Transactionを誰がどれだけ負担するかの分割情報。
- **Settlement**: 未精算残高を解消するための記録。
- **CategorySplitPreset**: カテゴリごとの分割プリセット。入力体験と一貫性を担保。

> **重要**: 金額・端数・split・settlement はコアドメイン。仕様変更は「影響範囲の明示」と「テスト同時更新」を必須とする。

---

## 5. 実装時の原則
- 既存構成・命名・責務分割を尊重する。
- 無関係なリファクタリングを混ぜない（差分は小さく、レビュー可能に）。
- 型安全を優先し、`any` は極力使わない。
- UI層へデータアクセスや計算ロジックをベタ書きしない。
- **Server Component / Client Component / Server Action / Repository / Service** の責務を分離する。
- Household / Ledger / Membership のスコープを壊さない。
- 将来の OCR / 銀行連携 / 税務対応を阻害する責務過多実装を避ける。

推奨フロー（最小）:
1. 要件をユースケースとスコープ（Household/Ledger）に分解
2. 先に型・入出力契約を定義
3. Repository/Serviceでロジック実装
4. UIは表示と操作に集中
5. Vitestで純粋ロジックを先に検証

---

## 6. データアクセス / 認証認可の原則
- Supabase利用時は **RLS前提** で設計する（アプリ側の条件分岐だけに依存しない）。
- Authセッションから現在ユーザーを解決し、Membership経由で権限を判定する。
- `owner / editor / viewer` の差異を常に意識する。
- Household外・Ledger外データを混在させる取得/更新を禁止する。
- 取得は必要最小限列を意識し、返却型を明示する。
- 失敗時は握りつぶさず、UIへ意味のあるエラーを返す。

---

## 7. UI 実装の原則
- モダンな見た目は維持しつつ、過剰演出より可読性・操作性を優先する。
- iOS / iPadOS / PWA を意識し、フォーム操作性（入力補助・誤操作防止）を重視する。
- 空状態・ローディング・エラー状態を必ず設計する。
- 金額入力・日付入力・分割編集は「誤入力しにくさ」を優先する。
- `preset / split / settlement` の計算はUIに埋め込まず、lib配下の純粋ロジックへ分離する。

---

## 8. テストの原則
- 純粋関数は Vitest で検証する。
- 最低限、**正常系 / 異常系 / 境界値** を含める。
- とくに以下を重点保護:
  - 金額計算
  - 端数処理
  - split計算
  - preset適用
  - settlement提案・相殺
- Supabase実接続不要部分は、マッピング・整形・集計関数として切り出してテストする。

## 9. PR / コミット / ドキュメントのルール
- **PRタイトルは日本語**
- **PR本文は日本語**
- **コミットメッセージは日本語**
- コードコメントは日本語を基本とする。
- UI文言は日本語を基本とする。
- README・開発者向け説明文も日本語優先。

PR本文に最低限含める項目:
- 背景/目的
- 変更内容（何をどこまで）
- 非変更内容（あえてやっていないこと）
- 影響範囲（Household/Ledger/Membership、RLS、金額計算）
- テスト内容

---

## 10. やってはいけないこと
- Household / Ledger / Membership の境界を曖昧にする変更
- RLS前提を無視したクエリ・更新
- 金額・端数処理の仕様をテスト更新なしで変更
- split / preset / settlement ロジックのUI直書き
- `any` 多用による型安全性の放棄
- 目的に無関係な大規模リネーム・整形のみコミット
- 一つのPRで複数目的を同時に達成しようとすること

---

## 11. 変更時のチェックリスト
- [ ] 変更目的は明確で、差分は最小か
- [ ] Household / Ledger / Membership スコープを壊していないか
- [ ] RLS・認可前提と矛盾していないか
- [ ] UIとロジックの責務分離ができているか
- [ ] 型が曖昧（`any`等）になっていないか
- [ ] 金額・端数・split・settlement への影響を確認したか
- [ ] 空/ロード中/失敗時UIを考慮したか
- [ ] owner/editor/viewer の権限差を確認したか
- [ ] Vitestで正常/異常/境界の観点を満たしたか
- [ ] PRに背景・影響範囲・テスト内容を日本語で記載したか

## 付記: このリポジトリでの実装位置の目安
- 画面: `app/`
- UIコンポーネント: `src/components/`
- データアクセス: `src/lib/repositories/`
- ドメインロジック: `src/lib/transactions/`, `src/lib/settlements/`, `src/lib/*service.ts`
- DB/RLS: `supabase/migrations/`, `supabase/policies/`
- テスト: `tests/`

この方針に従い、**「小さく安全に変更し、検証可能な状態で積み上げる」** ことを本プロジェクトの標準とします。
